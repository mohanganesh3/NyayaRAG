Section 1 — Master overview
Fundamental design principle and end-to-end data flow

Core principle: There is no single best RAG. There is only the best architecture for a given query type. NyayaRAG routes every query to its optimal pipeline. This routing decision is the most important engineering choice in the system.

Why multi-pipeline beats single-pipeline
Every existing Indian legal AI tool — BharatLaw, CaseMine, Lexis+ AI — uses one RAG pipeline for all query types. That is the root cause of their 17-34% hallucination rates. Consider three fundamentally different queries:

"What does Section 302 IPC say?" — needs BM25 for exact section number match. Dense retrieval alone misses exact legal terminology.
"How has the right to privacy developed in India?" — needs citation graph traversal across 70 years of judgments. Dense retrieval returns isolated cases with no doctrinal relationship.
"My landlord illegally locked me out, what can I do?" — needs hypothetical anchoring because the user has no legal vocabulary to anchor dense retrieval.
One pipeline cannot handle all three optimally. Multi-pipeline with routing is the only correct architecture.

Pipeline selection map
Pipeline	Primary trigger	Core mechanism	Target latency	Cost tier
1 — Hybrid RAG	Statutory lookup, specific case, default	BM25 + Dense + RRF + Cross-encoder + Authority rank	1.2–1.8s	Low
2 — GraphRAG	Doctrine evolution, constitutional law, multi-hop	Neo4j citation graph BFS traversal	3–5s	Medium
3 — CRAG layer	Applied ON TOP of Pipeline 1 for statutory queries	Relevance scoring + temporal validity check	+0.4s overhead	Low
4 — HyDE	Vague natural language, no legal terms	Generate hypothetical judgment → embed → retrieve	2–3s	Medium
5 — Agentic RAG	Document upload present	7-agent LangGraph system	15–30s	High

class QueryRouter:
    def analyze(self, query: str, context: CaseContext) -> QueryAnalysis:
        
        # Step 1: Extract sections and acts mentioned
        sections = self.section_extractor.extract(query)
        # section_extractor uses regex patterns:
        # IPC \d+[A-Z]? | CPC \d+ | CrPC \d+ | Article \d+
        # + BNS/BNSS/BSA equivalents

        entities = self.ner_model.extract(query)
        # Model: InLegalBERT fine-tuned for Indian legal NER
        # Entity types: PERSON, COURT, ACT, CASE_NAME, SECTION
        
        # Step 3: Classify query type
        # Primary classifier: rule-based (fast, deterministic)
        query_type = self._rule_classify(query, sections, entities)
        
        # Fallback: LLM classification if rules uncertain
        if query_type.confidence < 0.7:
            query_type = self._llm_classify(query)
        
        # Step 4: Extract jurisdiction
        jurisdiction = self._extract_jurisdiction(query, context)
        # Checks: explicit mention > case context court > default All India
        
        # Step 5: Check temporal flags
        post_july_2024 = self._check_new_codes(query, sections)
        # Any mention of IPC → flag for BNS mapping
        # Any date reference after July 1, 2024 → use BNS/BNSS
        
        # Step 6: Select pipeline
        pipeline = self._select_pipeline(query_type, context)
        
        return QueryAnalysis(...)
    
    def _select_pipeline(self, qt: QueryType, ctx: CaseContext) -> PipelineType:
        if ctx.has_uploaded_docs:
            return PipelineType.AGENTIC
        if qt in (MULTI_HOP_DOCTRINE, CONSTITUTIONAL, COMPARATIVE):
            return PipelineType.GRAPH_RAG
        if qt == VAGUE_NATURAL:
            return PipelineType.HYDE
        return PipelineType.HYBRID  # default for STATUTORY, CASE_SPECIFIC

Section 3 — Chunking & embedding strategy
Legal-aware document processing, BGE-M3 full specification, Qdrant collections

Why standard chunking fails for legal documents
Standard chunking strategies — fixed 512-token windows, recursive character splitting — break legal documents at the wrong boundaries. A judgment paragraph split mid-sentence loses the holding. A statute section split across two chunks loses the proviso that modifies the rule. Legal documents have semantic units that must be preserved.

Chunking strategy by document type
Document type	Chunk unit	Target size	Overlap	Reason
Judgment — headnotes	One headnote = one chunk	100–300 tokens	None	Each headnote is an independent legal proposition
Judgment — ratio decidendi	Paragraph-boundary, sentence-aware	300–512 tokens	50 tokens (1 sentence)	Core holding — never split a sentence
Judgment — obiter dicta	Paragraph-boundary	256–512 tokens	50 tokens	Less critical — can tolerate some boundary loss
Statute — section	One section = one chunk (always)	50–512 tokens	None	Section is atomic legal unit — never split across chunks
Statute — long section with sub-sections	Parent section + one sub-section	512 tokens max	Parent header repeated	Preserves context of which act/section the sub-section belongs to
Constitution — article	One article = one chunk	100–512 tokens	None	Each article is an independent constitutional provision
Law Commission report	Paragraph-boundary	512 tokens	100 tokens	Recommendations often span adjacent paragraphs

Stage 1 — BM25 sparse retrieval
BM25 is irreplaceable for legal text because lawyers use precise, domain-specific terminology. Section numbers, legal phrases, party names — these must be retrieved by exact or near-exact match. Dense retrieval alone fails here.

Implementation: rank-bm25 (Python, pure BM25 over full corpus text)

Index construction:
  - Tokenization: legal-aware tokenizer
    → preserves "498A" as one token (not split at digit)
    → preserves "force majeure" as compound
    → expands abbreviations: "SC" → "Supreme Court"
    → maps old codes: "IPC 302" → also indexed as "BNS 101"
  - IDF computed over full Indian legal corpus
  - Re-indexed: weekly (new judgments added)

Query expansion before BM25:
  1. Extract all section numbers mentioned
  2. Add BNS/BNSS/BSA equivalents automatically
  3. Add synonyms from legal thesaurus:
     "murder" → also query "culpable homicide", "Section 302"
     "bail" → also query "anticipatory bail", "Section 437", "BNSS 480"
  4. Add court-specific terminology variants

Retrieval:
  - top_k = 20
  - Score threshold: BM25 > 0.1 (filter pure noise)
  - Apply jurisdiction pre-filter:
    if jurisdiction_court specified:
      boost documents from binding courts × 1.5
  - Apply validity filter: current_validity == GOOD_LAW only
Stage 2 — Dense vector retrieval
Model: BGE-M3 (see Section 3)
Database: Qdrant

Query processing:
  1. If query is in Indian language:
     → translate to English via IndicTrans2
     → embed translated query
     → also embed original (multilingual BGE-M3 handles both)
     → take max similarity across both embeddings
  2. Embed normalized query
  3. Qdrant search with filters

Qdrant filter construction (applied before vector search):
  filters = {
    "must": [
      {"key": "current_validity", "match": {"value": "GOOD_LAW"}},
    ],
    "should": [
      # boost binding courts
      {"key": "jurisdiction_binding",
       "match": {"value": query.jurisdiction_court}},
    ],
    "must_not": [],
  }

  # Add practice area filter if high confidence classification
  if query.practice_area_confidence > 0.85:
    filters["must"].append({
      "key": "practice_area",
      "match": {"any": [query.practice_area.value]}
    })

  # Add date range filter if temporal query
  if query.time_sensitive and query.reference_date:
    filters["must"].append({
      "key": "date",
      "range": {"lte": query.reference_date.isoformat()}
    })

Search config:
  top_k = 20
  score_threshold = 0.6
  with_payload = True   # need metadata for downstream stages
  with_vectors = False  # don't return vectors (saves bandwidth)
Stage 3 — Reciprocal Rank Fusion (RRF)
Combines BM25 and dense results into a single ranked list. RRF is preferred over score normalization because scores from BM25 and dense retrieval are on incompatible scales.

def rrf_fusion(
    bm25_results: List[RankedDoc],
    dense_results: List[RankedDoc],
    k: int = 60
) -> List[RankedDoc]:
    """
    k=60 is the standard RRF parameter.
    Higher k = more weight on top results.
    Lower k = more democratic across rank positions.
    k=60 chosen after ablation on Indian legal benchmark.
    """
    scores: Dict[str, float] = {}
    
    for rank, doc in enumerate(bm25_results):
        scores[doc.chunk_id] = scores.get(doc.chunk_id, 0)
        scores[doc.chunk_id] += 1 / (k + rank + 1)
    
    for rank, doc in enumerate(dense_results):
        scores[doc.chunk_id] = scores.get(doc.chunk_id, 0)
        scores[doc.chunk_id] += 1 / (k + rank + 1)
    
    # Sort by fused score, return top 30
    fused = sorted(scores.items(), key=lambda x: x[1], reverse=True)
    return [doc_store.get(chunk_id) for chunk_id, _ in fused[:30]]

# Why top 30 → not top 8?
# Cross-encoder (Stage 4) is expensive per pair.
# RRF output is 30 candidates.
# Cross-encoder reranks to top 8.
# This gives high recall input to expensive reranker.
Stage 4 — Cross-encoder reranking
Model: cross-encoder/ms-marco-MiniLM-L-12-v2
       fine-tuned on Indian legal query-passage pairs

Fine-tuning data:
  - 10,000 (query, relevant_passage, irrelevant_passage) triplets
  - Source: CUAD + manually curated Indian legal Q&A
  - Training: 3 epochs, lr=2e-5, batch=16
  - Evaluation: nDCG@10 on held-out legal benchmark
  - Result: +18% nDCG@10 vs base cross-encoder on legal queries

Reranking process:
  Input: top-30 from RRF
  For each (query, chunk) pair:
    score = cross_encoder.predict([query, chunk.text])
    # Score: 0-1, probability of relevance
  
  Sort by cross-encoder score, take top 8
  
  Hard filter after reranking:
    Remove any chunk where:
      chunk.current_validity != GOOD_LAW
    → Ensures overruled cases never reach generator
    → Replaces with next-best valid chunk

Why 8 chunks to generator?
  - Legal queries have well-defined answers — more context
    adds noise, not signal
  - 8 × ~400 tokens = ~3,200 tokens of context
  - Leaves room in Claude's context for system prompt + output
  - Ablation showed 8 > 5 (+12% faithfulness), 8 ≈ 12 (minimal gain)
Stage 5 — Authority ranking (final sort within relevance tier)
After cross-encoder reranking, apply authority ordering
within each relevance tier (±0.05 score difference = same tier):

BINDING authority (for queried jurisdiction):
  Tier 1: Supreme Court — 9-judge Constitutional Bench
  Tier 2: Supreme Court — 5-judge bench
  Tier 3: Supreme Court — 3-judge bench
  Tier 4: Supreme Court — 2-judge bench (Division Bench)
  Tier 5: Supreme Court — single judge
  Tier 6: High Court of queried jurisdiction — Full Bench
  Tier 7: High Court of queried jurisdiction — Division Bench
  Tier 8: High Court of queried jurisdiction — single judge

PERSUASIVE authority (for queried jurisdiction):
  Tier 9: Other High Courts — sorted by citation count

Final output: top 5 binding + top 3 persuasive
Labeled in payload: "binding" | "persuasive" | "jurisdiction: [name]"
This label flows through to the output CitationBadge in UI.
Section 5 — Pipeline 2: GraphRAG
Citation graph construction, BFS traversal, doctrinal timeline, overruling detection

2
GraphRAG
Used for multi-hop doctrine queries and constitutional law. Traverses the citation graph to build a complete doctrinal timeline.
Why graph traversal is non-negotiable for doctrine queries
The right to privacy in India cannot be answered by retrieving the single most similar document. The correct answer requires tracing a doctrinal chain:

A.K. Gopalan (1950) → narrow reading of fundamental rights
         ↓ OVERRULED BY
Maneka Gandhi (1978) → due process expands Article 21 dramatically
         ↓ FOLLOWED BY
R. Rajagopal (1994) → privacy as component of Article 21
         ↓ FOLLOWED BY
PUCL v Union of India (1996) → telephone tapping violates privacy
         ↓ FOLLOWED BY
Puttaswamy (2017) — 9-judge bench → privacy is fundamental right

Dense retrieval returns one of these judgments.
GraphRAG returns the entire chain in chronological order.
The answer without the chain is incomplete.
Citation graph schema (Neo4j)
Node types:
  (:JudgmentNode {
    doc_id, citation, court, date,
    bench_size, ratio_decidendi,
    current_validity, practice_areas
  })
  
  (:StatuteNode {
    doc_id, act_name, section_number,
    current_text, is_in_force
  })
  
  (:ConceptNode {
    concept_id, name, area_of_law,
    description
    # Examples: "Right to Privacy", "Natural Justice",
    # "Double Jeopardy", "Promissory Estoppel"
  })

Edge types (directed):
  -[:CITES {citation_type}]->
    citation_type: 'follows' | 'distinguishes' | 'overrules'
                   'approves' | 'disapproves' | 'doubts'
                   'explains' | 'refers_to' | 'affirms'

  -[:INTERPRETS]->   (judgment → statute section)
  -[:APPLIES]->      (judgment → statute section)
  -[:EVOLVED_INTO]-> (old judgment → new judgment — same doctrine)
  -[:PART_OF]->      (judgment → concept node)

Scale targets:
  ~35K SC nodes
  ~500K HC nodes (sampled — not all HCs fully indexed in v1)
  ~15K statute nodes
  ~2M citation edges
  500 concept nodes (manually curated landmark case clusters)
GraphRAG pipeline — complete algorithm
def graph_rag_pipeline(query: QueryAnalysis) -> List[Chunk]:
    
    # Step 1: Identify anchor nodes
    # Pre-indexed: 500 landmark SC judgments tagged by concept
    anchors = self._find_anchors(query)
    
    # Anchor finding logic:
    # 1. Match query concepts against ConceptNode.name embeddings
    # 2. For constitutional queries: always include relevant
    #    Constitution Bench judgments as anchors
    # 3. For specific case queries: that case IS the anchor
    # 4. Fallback: top-3 most cited cases in practice area
    
    # Step 2: Bidirectional BFS from anchor nodes
    subgraph = self._traverse(anchors, query)
    
    # Step 3: Overruling detection and pruning
    pruned = self._prune_overruled(subgraph)
    
    # Step 4: PageRank within subgraph
    ranked = self._pagerank_subgraph(pruned)
    
    # Step 5: Build doctrinal timeline
    timeline = self._build_timeline(ranked, top_n=15)
    
    # Step 6: Fetch actual chunk text for each node
    chunks = self._fetch_chunks(timeline)
    
    return chunks

def _traverse(self, anchors, query, max_depth=4, max_nodes=50):
    """
    Bidirectional BFS traversal.
    
    Forward (what built on this?):
      Follow -[:CITES {type: 'follows'}]-> edges forward
      Follow -[:EVOLVED_INTO]-> edges
      Collect all judgments that followed the anchor
    
    Backward (what did this build on?):
      Follow -[:CITES]-> edges backward (reverse direction)
      Collect foundational cases in the doctrine
    
    Filtering during traversal:
      - Only follow edges where target.current_validity == GOOD_LAW
      - Do NOT traverse through OVERRULED nodes
        (they are dead ends in the doctrine graph)
      - Weight by: court_level × recency × citation_count
    
    Early stopping:
      - If max_depth reached: stop
      - If max_nodes reached: keep highest-weight nodes
      - If all frontier nodes are OVERRULED: stop
    """

def _prune_overruled(self, subgraph):
    """
    Critical step — removes reversed doctrine.
    
    For each node in subgraph:
      If node has incoming -[:CITES {type: 'overrules'}]-> edge:
        Mark as OVERRULED_IN_SUBGRAPH
        Do NOT include in output
        DO include the overruling case (it's the current position)
        Log: "Note: [X] overruled by [Y] — using [Y]"
    
    This is what prevents citing Gopalan (1950) when answering
    questions about Article 21 — it was overruled by Maneka Gandhi.
    """

def _build_timeline(self, nodes, top_n=15):
    """
    Orders nodes into: foundational → development → current
    
    Sort by date ascending
    Group into phases:
      FOUNDATIONAL: early cases establishing the doctrine
      DEVELOPMENT: cases refining and extending
      CURRENT: most recent authoritative statement
    
    Always include:
      - The overruling judgment (if any) prominently
      - The most recent Constitution Bench judgment
      - Any judgment from last 3 years (current position)
    
    Output: ordered sequence with phase labels
    This structure is passed to the generator as structured context.

Section 6 — Pipeline 3: CRAG

3
CRAG — Corrective RAG
Applied as a validation layer on top of Pipeline 1 for all statutory queries. Also runs as a quality gate for all other pipelines.
CRAG is not a standalone pipeline. It is a validation layer that wraps Hybrid RAG (Pipeline 1) for statutory queries, and runs as a quality gate after retrieval in all other pipelines. Think of it as the immune system of the retrieval layer.

CRAG validation — complete implementation
class CRAGValidator:
    
    def validate(
        self,
        query: str,
        retrieved_chunks: List[Chunk]
    ) -> CRAGResult:
        
        # Score 1: Semantic relevance
        # Does the retrieved context semantically answer the query?
        relevance_scores = [
            self.relevance_model.score(query, chunk.text)
            for chunk in retrieved_chunks
        ]
        avg_relevance = mean(relevance_scores)
        
        # Score 2: Coverage check
        # Are all key entities from the query present in retrieved chunks?
        query_entities = self.ner.extract(query)
        chunk_text = " ".join(c.text for c in retrieved_chunks)
        entity_coverage = self._check_entity_coverage(
            query_entities, chunk_text
        )
        
        # Combined score
        crag_score = 0.7 * avg_relevance + 0.3 * entity_coverage
        
        if crag_score > 0.70:
            return CRAGResult(
                action='PROCEED',
                score=crag_score,
                chunks=retrieved_chunks
            )
        
        elif crag_score > 0.40:
            # Partial relevance: decompose query and re-retrieve
            sub_queries = self._decompose_query(query)
            # sub_queries: break complex query into 2-3 focused questions
            new_chunks = []
            for sq in sub_queries:
                new_chunks.extend(self.hybrid_pipeline.retrieve(sq))
            # Deduplicate and rerank
            deduped = self._dedup(new_chunks)
            return CRAGResult(
                action='REFINED',
                score=crag_score,
                chunks=deduped,
                refinement_note='Query decomposed for better retrieval'
            )
        
        else:
            # Very low relevance: corpus may not contain answer
            # Escalate to web search for very recent developments
            web_results = self._search_web(query)  # limited to legal sites
            if web_results:
                return CRAGResult(
                    action='WEB_SUPPLEMENTED',
                    score=crag_score,
                    chunks=retrieved_chunks + web_results,
                    warning='Limited corpus coverage — web sources included'
                )
            else:
                return CRAGResult(
                    action='INSUFFICIENT',
                    score=crag_score,
                    chunks=retrieved_chunks,
                    warning='Insufficient information in corpus for this query. Please verify with primary sources.'
                )
Temporal validation — the most critical check
Indian law changes constantly. BNS replaced IPC in July 2024. Dozens of sections are amended every year. Any system that doesn't check temporal validity is serving outdated law.

def validate_temporal_validity(
    chunk: DocumentChunk
) -> TemporalValidationResult:
    
    if chunk.doc_type == 'statute':
        statute = db.get_statute(chunk.doc_id)
        
        # Check 1: Is the Act itself still in force?
        if not statute.current_validity:
            return TemporalValidationResult(
                valid=False,
                severity='CRITICAL',
                reason=f"Act repealed. Replaced by: {statute.replaced_by}",
                replacement_doc_id=statute.replacement_doc_id
            )
        
        # Check 2: Has this specific section been amended?
        section = statute.get_section(chunk.section_number)
        if section.amendments:
            latest_amendment = max(
                section.amendments,
                key=lambda a: a.effective_date
            )
            if latest_amendment.effective_date > chunk.embedded_at:
                return TemporalValidationResult(
                    valid=False,
                    severity='IMPORTANT',
                    reason=f"Section amended on {latest_amendment.effective_date}",
                    current_text=section.current_text,
                    amendment_details=latest_amendment
                )
        
        # Check 3: IPC/CrPC/Evidence Act — post July 2024 queries
        if (chunk.act_name in ['IPC', 'CrPC', 'Indian Evidence Act']
            and query.reference_date >= date(2024, 7, 1)):
            bns_equivalent = BNS_MAP.get(
                f"{chunk.act_name} {chunk.section_number}"
            )
            return TemporalValidationResult(
                valid=True,  # Old code still valid for pre-2024 offences
                warning=f"New equivalent: {bns_equivalent} (BNS/BNSS)",
                bns_section=bns_equivalent,
                note="Use BNS section for offences committed after July 1, 2024"
            )
        
        return TemporalValidationResult(valid=True)
    
    elif chunk.doc_type == 'judgment':
        # Check: has this judgment been overruled since indexing?
        validity = db.get_current_validity(chunk.doc_id)
        if validity == ValidityStatus.OVERRULED:
            overruling = db.get_overruling_case(chunk.doc_id)
            return TemporalValidationResult(
                valid=False,
                severity='CRITICAL',
                reason=f"Overruled by {overruling.citation}",
                replacement_doc_id=overruling.doc_id
            )
        return TemporalValidationResult(valid=True)
Section 7 — Pipeline 4: HyDE
Hypothetical Document Embeddings for vague natural language queries

4
HyDE — Hypothetical Document Embeddings
Used for vague natural language queries from non-expert users. Generates a hypothetical ideal judgment as a retrieval anchor.
Why HyDE solves the vocabulary gap problem
A junior advocate or a lay person asks: "My landlord changed the locks without notice, what are my rights?" Dense retrieval fails because this query has no legal vocabulary. The relevant cases use terms like "illegal dispossession", "Section 6 Specific Relief Act", "forcible eviction" — none of which appear in the query.

HyDE solves this by generating a hypothetical ideal answer first, then using that answer (which contains legal vocabulary) as the retrieval query.

def hyde_pipeline(query: QueryAnalysis) -> List[Chunk]:
    
    # Step 1: Generate hypothetical ideal judgment excerpt
    hypothetical = llm.generate(
        system="""You are a senior Indian advocate with 20 years of experience.
        Write a 120-word excerpt from a hypothetical Supreme Court or High Court
        judgment that would perfectly answer the user's question.
        Rules:
        - Use real Indian legal terminology and section numbers
        - Reference the correct Acts (Specific Relief Act, Transfer of Property Act, etc.)
        - Do NOT fabricate real case citations
        - Write in judicial prose style
        - Include the legal principle being applied
        """,
        user=f"User's question: {query.raw_query}"
    )
    # Example output for landlord query:
    # "The petitioner's right to possession cannot be disturbed
    #  without due process of law. Section 6 of the Specific Relief
    #  Act, 1963 provides that any person dispossessed of immovable
    #  property without consent may recover possession by suit.
    #  The principle of nemo debet esse judex in propria causa..."
    
    # Step 2: Embed the hypothetical (not the original query)
    hyp_embedding = bge_m3.encode(hypothetical)
    
    # Step 3: Use hypothetical embedding for dense retrieval
    dense_results = qdrant.search(
        collection="sc_judgments",
        query_vector=hyp_embedding,
        top_k=20,
        query_filter={"must": [{"key": "current_validity",
                                "match": {"value": "GOOD_LAW"}}]}
    )
    
    # Step 4: ALSO do BM25 on original query
    # HyDE alone misses exact terminology the user DID provide
    bm25_results = bm25_index.search(query.raw_query, top_k=10)
    
    # Step 5: RRF fusion of both results
    fused = rrf_fusion(dense_results, bm25_results)
    
    # Step 6: Rerank with cross-encoder using ORIGINAL query
    # (not hypothetical — cross-encoder should score against user intent)
    reranked = cross_encoder.rerank(query.raw_query, fused, top_k=8)
    
    # Step 7: Standard CRAG validation
    crag_result = crag_validator.validate(query.raw_query, reranked)
    
    return crag_result.chunks

Section 9 — Zero hallucination architecture
Type 1 (fabrication) and Type 2 (misgrounding) — complete implementation

Stanford 2025: Westlaw AI hallucinates 34% of the time. Lexis+ AI 17%. These are $500M products. The cause is not bad prompting — it is architectural. RAG without structural citation constraints will hallucinate. NyayaRAG prevents hallucination structurally, not with prompts.

Type 1: Citation fabrication — LLM invents non-existent case
The most common failure. The LLM writes "the Supreme Court held in Sharma v Union of India (2019) 4 SCC 210..." — this case does not exist.

Prevention: Constrained generation with placeholder protocol

System prompt (given to Claude Sonnet for generation):
"""
You are a legal research assistant. Answer the query using ONLY
the provided context chunks. 

CITATION RULES — NON-NEGOTIABLE:
1. When you need to cite a case: write [CITE: brief description]
   Example: [CITE: SC case on bail in murder cases, 2-judge bench]
   NEVER write an actual case name or citation yourself.

2. When you need to cite a statute: write [STATUTE: Act name, Section]
   Example: [STATUTE: IPC, Section 302]
   NEVER write the section text yourself — it will be fetched.

3. If the context does not support a legal proposition:
   Write [UNSUPPORTED: proposition text]
   Do NOT omit it — mark it explicitly.

The system will replace all placeholders with verified citations.
"""

Post-generation: CitationResolver
  Input: generated text with [CITE: ...] placeholders
  
  For each placeholder:
    1. Match against retrieved_chunks by semantic similarity
       if similarity > 0.85 → VERIFIED → replace with official citation
    
    2. If no chunk match → search full corpus
       if corpus match score > 0.80 → VERIFIED → use corpus citation
    
    3. If no match found:
       status = 'UNVERIFIED'
       action = 'SHOW_WITH_WARNING'  ← never block, never omit
       output: legal proposition shown with blue [UNVERIFIED] badge
       message: "Legal principle identified — specific case not located
                 in corpus. Verify with primary sources."
    
    NEVER happens: no hallucinated citation reaches the output.
    Either verified → real citation, or unverified → explicit warning.
Type 2: Misgrounding — real case cited incorrectly
More dangerous than fabrication. The case exists. The citation is real. But it doesn't say what the answer claims. This is what Stanford's study found most prevalent.

def check_misgrounding(
    claim: str,          # sentence from generated answer
    doc_id: str,         # the cited case/statute doc_id
    citation: str        # the citation string
) -> MisgroundingResult:
    
    # Step 1: Fetch source document
    source_doc = corpus.get(doc_id)
    
    # Step 2: Extract relevant passage from source
    # (not the entire document — find the passage closest to the claim)
    relevant_passages = retrieve_within_doc(
        query=claim,
        doc_id=doc_id,
        top_k=3
    )
    best_passage = relevant_passages[0].text
    
    # Step 3: Semantic similarity check
    claim_embedding = bge_m3.encode(claim)
    passage_embedding = bge_m3.encode(best_passage)
    sim = cosine_similarity(claim_embedding, passage_embedding)
    
    # Step 4: NLI entailment check
    # Does the passage ENTAIL (logically support) the claim?
    nli_result = nli_model.predict(
        premise=best_passage,
        hypothesis=claim
    )
    # nli_model: cross-encoder/nli-deberta-v3-base
    # (free, HuggingFace, strong NLI performance)
    
    # Step 5: Decision
    if sim > 0.82 and nli_result.label == 'ENTAILMENT':
        return MisgroundingResult(
            status='VERIFIED',
            confidence=sim,
            source_passage=best_passage
        )
    
    elif sim > 0.55 or nli_result.label == 'NEUTRAL':
        return MisgroundingResult(
            status='UNCERTAIN',
            confidence=sim,
            source_passage=best_passage,
            action='SHOW_SOURCE_TO_USER',
            # UI: automatically expand source panel showing this passage
            # User can judge whether it supports the claim
        )
    
    else:
        # Genuine misgrounding detected
        return MisgroundingResult(
            status='MISGROUNDED',
            confidence=sim,
            action='REMOVE_AND_RERETRIEVE',
            # Remove this citation from output
            # Trigger re-retrieval for this specific claim
            # If re-retrieval also fails → mark UNVERIFIED
        )
Appeal chain validator — prevents citing reversed judgments
def validate_appeal_chain(doc_id: str) -> AppealValidationResult:
    doc = corpus.get(doc_id)
    
    if not doc.appeal_history:
        return AppealValidationResult(valid=True, use_doc_id=doc_id)
    
    # Find the final authoritative node in the appeal chain
    final_nodes = [n for n in doc.appeal_history if n.is_final_authority]
    
    if not final_nodes:
        # Appeal pending — not final
        return AppealValidationResult(
            valid=True,  # can still cite pending cases
            warning='Appeal pending — this may not be the final judgment',
            use_doc_id=doc_id
        )
    
    final = final_nodes[-1]  # most recent final authority
    
    if final.outcome == 'reversed':
        return AppealValidationResult(
            valid=False,
            severity='CRITICAL',
            warning=f'This judgment was REVERSED by {final.citation}',
            use_doc_id=final.doc_id,  # redirect to the reversing judgment
            show_reversal_banner=True
            # UI: red banner "This judgment was reversed. Showing reversal."
        )
    
    if final.outcome == 'modified':
        return AppealValidationResult(
            valid=True,  # can cite, but with modification note
            warning=f'Modified by {final.citation} on [{final.modified_aspects}]',
            use_doc_id=doc_id,
            supplementary_doc_id=final.doc_id
        )
    
    return AppealValidationResult(valid=True, use_doc_id=doc_id)
